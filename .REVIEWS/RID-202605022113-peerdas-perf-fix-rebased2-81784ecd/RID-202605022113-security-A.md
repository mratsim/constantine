## Detailed Analysis

### batchAffine → batchAffine_vartime Migration

The most prominent change across the diff is the systematic replacement of `batchAffine` (constant-time Montgomery batch inversion) with `batchAffine_vartime` (variable-time). New `batchAffine_vartime` implementations were added for both `EC_ShortW` (Short Weierstrass, Jacobian and Projective) and `EC_TwEdw` (Twisted Edwards) curves.

**Affected call sites:**
- `kzg.nim:363` — `kzg_verify_batch`: converts `commits_min_evals` (public commitment differences)
- `kzg_parallel.nim:170` — `kzg_verify_batch_parallel`: same public data
- `kzg_multiproofs.nim:828` — `kzg_coset_prove`: converts proof output (public)
- `kzg_multiproofs.nim:709` — polyphase spectrum batch conversion from Jacobian→Affine (public SRS data)
- `eth_verkle_ipa.nim:225,260,649` — IPA proof generation (public proof elements)
- `ec_scalar_mul_vartime.nim:398,420` — wNAF precomputation table conversion (already in a `*_vartime` function)
- `ec_shortweierstrass_batch_ops_parallel.nim:224` — parallel sum reduction (public)

**Assessment:** All these code paths process **public** data — KZG proofs, commitments, polyphase spectra, and IPA proof elements. None of these contain secret-dependent values. Per the side-channel guidance:

> Only flag side-channel concerns when the algorithm actually processes secret-dependent data paths.
> zkVMs, data availability sampling (DAS), public verification algorithms, non-secret cryptographic primitives — if the code only processes public data, timing side-channels are NOT a security concern.

The `scalarMul_wNAF_vartime` and `scalarMulEndo_wNAF_vartime` functions already process secret scalars in variable-time (the scalar bits determine loop behavior), so replacing the batch conversion within them with a vartime variant is consistent and does not introduce new leakage.

### ToeplitzAccumulator — New MSM-based Toeplitz Implementation

The `ToeplitzAccumulator` type replaces `toeplitzMatVecMulPreFFT`. Key security-relevant observations:

1. **Pointer casts:** `finish()` reinterprets `scratchScalars` (typed as `ptr UncheckedArray[F]`) as `ptr UncheckedArray[F.getBigInt()]`. This is safe because a static assertion `sizeof(F) == sizeof(F.getBigInt())` guards the cast, and `fromField` is used for type conversion within the buffer.

2. **`addr ctx.points[i * ctx.L]` cast:** In `finish()`, the pointer arithmetic is bounded: `i` ranges `0..n-1`, `L` is the accumulation count, and the buffer is `size * L` elements. Maximum offset is `(n-1) * L < n * L`, so no out-of-bounds access.

3. **Resource cleanup:** The `check`/`checkReturn` templates with labeled `block HappyPath:` ensure all `allocHeapAligned` allocations are freed on any failure path. Verified that `vExtFftAff`, `ifftResult`, and `vExt` are all freed.

4. **`=destroy` correctness:** Nil-checks on all pointer fields prevent double-free. The `init()` proc also defensively frees existing allocations before allocating new ones.

### computeAggRandScaledInterpoly — Assertion-Only Validation

The function changed from returning `bool` (returning `false` on invalid input) to `void` with `doAssert` for all parameter checks. The call chain was traced:

1. `eth_eip7594_peerdas.nim:495-497` validates `cell_indices` against `CELLS_PER_EXT_BLOB` before they become `evalsCols`.
2. `kzg_coset_verify_batch` (lines 653-664) validates all array lengths and element bounds (`commitmentIdx`, `evalsCols`) against their respective containers before calling `computeAggRandScaledInterpoly`.
3. The `doAssert` in `computeAggRandScaledInterpoly` serves as internal invariant enforcement, not primary defense.

**Assessment:** The caller-level validation remains intact and comprehensive. The assertion-only change is safe because it's an internal function with validated inputs from the public API.

### bit_reversal_permutation Aliasing Support

New `bit_reversal_permutation_noalias` (non-aliasing variant) and aliasing-aware `bit_reversal_permutation(dst, src)` that detects `dst[0].addr == src[0].addr` and uses a temporary buffer. In-place `bit_reversal_permutation(buf)` delegates to the noalias variant. Correct and complete.

### Positive Changes

- **Alias-aware bit_reversal_permutation:** Fixes potential data corruption when dst and src reference the same array, which was silently broken in the previous unaliased implementation.
- **Proper error handling in ToeplitzAccumulator:** The `check`/`checkReturn` pattern with labeled cleanup blocks eliminates the manual error-checking sprawl from `toeplitzMatVecMulPreFFT` and ensures all allocations are freed on any error path.
- **Defensive init:** `ToeplitzAccumulator.init()` frees existing allocations before creating new ones, preventing leaks on accidental double-init.
- **N<=0 early return in batchAffine/batchAffine_vartime:** Prevents invalid memory access with zero-length batches.
