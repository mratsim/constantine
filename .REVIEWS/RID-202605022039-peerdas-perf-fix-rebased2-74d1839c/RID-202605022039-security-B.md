### [SECURITY] SEC-B-001: Error model change — `computeAggRandScaledInterpoly` drops runtime validation in release builds - constantine/commitments/kzg_multiproofs.nim:502-579

**Location:** constantine/commitments/kzg_multiproofs.nim:502-579
**Severity:** Medium
**Confidence:** 0.8

**Diff Under Review:**
```diff

        interpoly: var PolynomialCoef[L, Fr[Name]],
        evals: openArray[array[L, Fr[Name]]],
        evalsCols: openArray[int],
        domain: FrFFT_Descriptor[Fr[Name]],

+      N: static int) {.meter.} =
   ...
-  if evals.len != evalsCols.len or linearIndepRandNumbers.len < evalsCols.len:
-    return false
+  doAssert evals.len == evalsCols.len, "Internal error: evals and evalsCols must have same length"
+  doAssert linearIndepRandNumbers.len >= evalsCols.len, "Internal error: linearIndepRandNumbers must cover all evals"
   ...

-      return false
+    doAssert c >= 0 and c < NumCols, "Internal error: Column index out of bounds: " & $c

```

And the call site:
```diff
-  if not interpoly.computeAggRandScaledInterpoly(
+  interpoly.computeAggRandScaledInterpoly(
     evals, evalsCols, domain, linearIndepRandNumbers, N
-  ):
-    return false
+  )
```

**Issue: Defense-in-depth erosion on validation path**

The function `computeAggRandScaledInterpoly` previously returned `bool` and performed runtime validation of:
1. Length equality: `evals.len == evalsCols.len`
2. Coverage: `linearIndepRandNumbers.len >= evalsCols.len`
3. Bounds: `c >= 0 and c < NumCols` for each column index

These were replaced with `doAssert` calls, which are compiled away in release builds (Nim's default `-d:release` disables assertions). In such builds, invalid inputs would lead to:
- Out-of-bounds memory reads on `evalsCols[k]` beyond `evals.len`
- Out-of-bounds heap indexing into `agg_cols[col]` with negative or oversized column indices
- Memory corruption and potential denial of service

**Mitigating factors:**
- The sole caller `kzg_coset_verify_batch` (lines 652-664) performs equivalent `if ... return false` validation before calling `computeAggRandScaledInterpoly`
- `computeAggRandScaledInterpoly` is not exported (`*`), so direct external calls require import from within the same module
- In practice, the caller's validation provides complete coverage

**Risk assessment:**
While the current caller validates all inputs, this is a defense-in-depth regression. If a future caller omits validation or the validation logic diverges, the function becomes silently unsafe in release builds. The function name suggests "internal error" but the inputs (`evals`, `evalsCols`, `linearIndepRandNumbers`) flow from `kzg_coset_verify_batch` which accepts them from external verification inputs.

**Suggested Change:** Either (a) keep the `bool` return type and `if ... return false` guards, or (b) if the function is truly internal-only, document the precondition contract explicitly and use `raise newException(ValueError, ...)` instead of `doAssert` for runtime safety.

**Issue: Timing side-channel assessment of batchAffine_vartime across all call sites**

The diff introduces `batchAffine_vartime`, which explicitly branches on `z == 0` (infinity point detection) using `if zero(i).bool():` instead of the constant-time `SecretBool` masking used in `batchAffine`. This creates a timing variation dependent on the z-coordinate distribution of input points.

**Analysis of each call site:**

1. **`kzg_verify_batch` / `kzg_verify_batch_parallel`** (kzg.nim:363, kzg_parallel.nim:170):
   - Points are `commitments[i] - evals_at_challenges[i] * G`, where commitments and evaluations are public
   - z-coordinates are publicly determined; timing leak reveals nothing secret
   - ✅ Safe

2. **`ipa_prove`** (eth_verkle_ipa.nim:225, 260, 649):
   - IPA proving is the sender-side operation; polynomial being proven is public
   - z-coordinates of intermediate points are derived from public commitments
   - ✅ Safe

3. **`computePolyphaseDecompositionFourier`** (kzg_multiproofs.nim:364):
   - Converts polyphase spectrum bank from Jacobian to affine
   - Source data is the trusted setup SRS (public constants)
   - ✅ Safe

4. **`kzg_coset_prove`** (kzg_multiproofs.nim:458):
   - Converts IFFT output (public) to affine for proof output
   - z-coordinates derive from public polynomial coefficients and SRS
   - ✅ Safe

5. **`scalarMul_wNAF_vartime` / `scalarMulEndo_wNAF_vartime`** (ec_scalar_mul_vartime.nim:362, 430):
   - Precomputation table: `[k]P` for k=0..precompSize, where P is the input point
   - z-coordinates depend only on P (public), not on the scalar
   - The secret-dependent NAF encoding step is already variable-time (entire function is `{.tags:[VarTime]}`)
   - The function explicitly documents: "MUST NOT be used with secret data"
   - ✅ Safe (no new information leaked)

6. **`sum_reduce_vartime_parallelChunks`** (ec_shortweierstrass_batch_ops_parallel.nim:66):
   - Parallel reduction of EC points; z-coordinates are intermediate computation results
   - No secret-dependent data flows through this path
   - ✅ Safe

**Conclusion:** None of the `batchAffine_vartime` call sites process secret-dependent z-coordinates. The timing variation leaks only information about public data, which is not exploitable. The side-channel guidance explicitly excludes "timing side-channels on internal code paths that do NOT process secrets."

### [SECURITY] SEC-B-003: `ToeplitzAccumulator` scratch buffer reinterpret-cast relies on static size invariant - constantine/math/matrix/toeplitz.nim:281-295

**Location:** constantine/math/matrix/toeplitz.nim:281-295
**Severity:** Informational
**Confidence:** 1.0

**Diff Under Review:**
```nim
  # Invariant: scratchScalars is typed as F but re-interpreted as F.getBigInt() below.
  # This requires sizeof(F) == sizeof(F.getBigInt()), which holds for all production
  # field types (e.g. Fr[BLS12_381] is 32 bytes in both representations).
  static: doAssert sizeof(F) == sizeof(F.getBigInt()), "scratchScalars cast requires sizeof(F) == sizeof(F.getBigInt())"

  let scalars = cast[ptr UncheckedArray[F.getBigInt()]](ctx.scratchScalars)
```

**Issue: Fragile memory reinterpretation pattern**

The `ToeplitzAccumulator.finish` method allocates `scratchScalars` as `ptr UncheckedArray[F]` (field elements) but reinterprets it as `ptr UncheckedArray[F.getBigInt()]` (big integers) via `cast`. This relies on `sizeof(F) == sizeof(F.getBigInt())` being true at compile time.

For `Fr[BLS12_381]` this holds (both 32 bytes), and the `static: doAssert` provides compile-time checking. However:
- This pattern is fragile: adding a new field type where sizes differ would cause silent memory corruption at runtime (the static check catches it, but only if the type is ever instantiated in a compiled binary)
- The `cast` bypasses Nim's type system entirely — no bounds checking on the reinterpreted array
- If `F.getBigInt()` has padding or different alignment requirements than `F`, the reinterpretation could cause alignment faults

This is not an active vulnerability but represents a maintenance risk in the codebase.

**Suggested Change:** Consider using `union` types or explicit `reinterpret` helpers that document the size/alignment contract, rather than raw `cast`.

## Positive Changes

1. **`bit_reversal_permutation` aliasing fix** (fft_common.nim:307-321): The new aliasing detection in `bit_reversal_permutation` prevents data corruption when `dst` and `src` are the same array — a correctness issue that could have led to silently wrong cryptographic computations. The fix uses a temporary buffer and `copyMem` for the aliased case.

2. **`ToeplitzAccumulator` memory safety** (toeplitz.nim:201-211): The type implements `=destroy` with nil-checks on all ptr fields, sets pointers to nil after freeing, and declares `=copy {.error.}` to prevent accidental copies. The `init` method defensively frees existing allocations before reallocating, preventing double-init bugs.

3. **Error path resource cleanup** (toeplitz.nim:355-378): The `toeplitzMatVecMul` procedure uses labeled `block HappyPath:` with the `check` template for structured cleanup, ensuring all heap allocations are freed on any error path — eliminating a class of memory leaks.

4. **`batchAffine` N≤0 guards** (ec_shortweierstrass_batch_ops.nim:30-32, 98-100; ec_twistededwards_batch_ops.nim:28-30): Both `batchAffine` and `batchAffine_vartime` now include `if N <= 0: return` guards at the start, preventing division-by-zero or out-of-bounds access on empty batches. This is a robustness improvement over the previous unguarded code.
