### [QA] QA-001: New public `batchAffine_vartime*` (Short Weierstrass) functions lack `##` docstrings

**Location:** `constantine/math/elliptic/ec_shortweierstrass_batch_ops.nim:185-250`, `262-334`
**Severity:** Medium
**Confidence:** 1.0

**Diff Under Review:**
```diff
+func batchAffine_vartime*[F, G](
+       affs: ptr UncheckedArray[EC_ShortW_Aff[F, G]],
+       projs: ptr UncheckedArray[EC_ShortW_Prj[F, G]],
+       N: int) {.tags:[VarTime], meter.} =
+  if N <= 0:
+    return
+
+  # Algorithm: Montgomery's batch inversion
+  # - Speeding the Pollard and Elliptic Curve Methods of Factorization
+  #   Section 10.3.1
+  #   Peter L. Montgomery
...
+func batchAffine_vartime*[F, G](
+       affs: ptr UncheckedArray[EC_ShortW_Aff[F, G]],
+       jacs: ptr UncheckedArray[EC_ShortW_Jac[F, G]],
+       N: int) {.tags:[VarTime], meter.} =
+  if N <= 0:
+    return
+
+  # Algorithm: Montgomery's batch inversion
...
```

**Issue:** **No `##` docstrings on exported (`*`) public API functions.**

Both the projective and Jacobian variants of `batchAffine_vartime*` are exported public API functions with no `##` docstring. They have algorithm citations (Montgomery, Brent-Zimmermann) embedded as code comments (`#`), which is valuable for maintainers, but there is no user-facing `##` docstring describing:

- The function's purpose (batch projective/Jacobian → affine conversion, variable-time)
- Parameter descriptions (`affs`, `projs`/`jacs`, `N`)
- Return behavior (in-place, no return value)
- The `VarTime` security implication (not constant-time; do not use with secret coordinates)

This contrasts with other public functions in the codebase (e.g., `toeplitzMatVecMul` at line 311 which has a comprehensive `##` docstring with `@param` annotations and `@return`).

**Suggested Change:** Add `##` docstrings following the project's convention:

```nim
func batchAffine_vartime*[F, G](
       affs: ptr UncheckedArray[EC_ShortW_Aff[F, G]],
       projs: ptr UncheckedArray[EC_ShortW_Prj[F, G]],
       N: int) {.tags:[VarTime], meter.} =
  ## **Variable-time** batch conversion from projective to affine coordinates.
  ##
  ## Uses Montgomery's batch inversion to perform N coordinate conversions
  ## with a single field inversion. Variable-time — do not use with secret coordinates.
  ##
  ## @param affs: Output array of affine points (pre-allocated, length N)
  ## @param projs: Input array of projective points (length N)
  ## @param N: Number of points to convert
  ##
  ## Algorithm: Montgomery's batch inversion
  ## - Speeding the Pollard and Elliptic Curve Methods of Factorization
  ...
```

### [QA] QA-004: `ToeplitzAccumulator.init` has no docstring

**Location:** `constantine/math/matrix/toeplitz.nim:213-248`
**Severity:** Medium
**Confidence:** 1.0

**Diff Under Review:**
```diff
+proc init*[EC, ECaff, F](
+  ctx: var ToeplitzAccumulator[EC, ECaff, F],
+   frFftDesc: FrFFT_Descriptor[F],
+   ecFftDesc: ECFFT_Descriptor[EC],
+  size: int,
+  L: int
+): ToeplitzStatus {.raises: [], meter.} =
+  # Free existing allocations (defensive: handles accidental double-init)
...
```

**Issue:** **Public `init` proc with no docstring for a significant memory allocator.**

`ToeplitzAccumulator.init` is an exported public function that allocates ~772 KB of heap memory for production parameters (L=64, CDS=128). It has no docstring explaining:

- What it does (allocates transposed buffers for `size * L` coefficients and points)
- Parameter meaning (`size` = CDS, `L` = number of accumulations)
- Return value (success status)
- Memory layout (transposed storage for cache-efficient MSM)
- The defensive double-init behavior mentioned in the comment

The `accumulate` and `finish` methods have brief docstrings, but the init function that sets up the entire data structure is undocumented.

**Suggested Change:**

```nim
proc init*[EC, ECaff, F](
  ctx: var ToeplitzAccumulator[EC, ECaff, F],
   frFftDesc: FrFFT_Descriptor[F],
   ecFftDesc: ECFFT_Descriptor[EC],
  size: int,
  L: int
): ToeplitzStatus {.raises: [], meter.} =
  ## Initialize the Toeplitz accumulator with pre-allocated transposed buffers.
  ##
  ## Allocates three aligned buffers:
  ## - `coeffs`: [size*L] field elements (transposed FFT results)
  ## - `points`: [size*L] affine EC points
  ## - `scratchScalars`: [max(size, L)] working buffer
  ##
  ## Safe to call multiple times — existing allocations are freed first.
  ##
  ## @param ctx: Accumulator to initialize
  ## @param frFftDesc: Field FFT descriptor (order >= size)
  ## @param ecFftDesc: EC FFT descriptor (order >= size)
  ## @param size: FFT/MSM size per accumulate step (must be power of two)
  ## @param L: Number of accumulate steps before finish
  ## @return: Toeplitz_Success on allocation, Toeplitz_SizeNotPowerOfTwo if invalid
```

### [QA] QA-005: Comment inconsistency between projective and Jacobian `batchAffine_vartime` overloads

**Location:** `constantine/math/elliptic/ec_shortweierstrass_batch_ops.nim:202-203` vs `279-280`
**Severity:** Low
**Confidence:** 0.9

**Diff Under Review:**
```diff
  # Projective overload (line 202-203):
+  # To avoid temporaries, we store partial accumulations
+  # in affs[i].x and affs[i].y will store if the input was 0

  # Jacobian overload (line 279-280):
+  # To avoid temporaries, we store partial accumulations
+  # in affs[i].x and whether z == 0 in affs[i].y
```

**Issue:** **Comment wording drifted between sibling overloads.**

The projective overload says "affs[i].y **will store if the input was 0**" while the Jacobian overload says "**whether z == 0** in affs[i].y". Both describe the same mechanism (storing the zero-detection flag for `projs[i].z` / `jacs[i].z`), but the phrasing is inconsistent.

**Suggested Change:** Use consistent wording. The Jacobian version ("whether z == 0 in affs[i].y") is slightly more precise. Update the projective overload to match:

```nim
# To avoid temporaries, we store partial accumulations
# in affs[i].x and whether z == 0 in affs[i].y
```

### [QA] QA-007: `ToeplitzAccumulator.finish` docstring is terse

**Location:** `constantine/math/matrix/toeplitz.nim:276`
**Severity:** Low
**Confidence:** 1.0

**Diff Under Review:**
```diff
+proc finish*[EC, ECaff, F](
+  ctx: var ToeplitzAccumulator[EC, ECaff, F],
+  output: var openArray[EC]
+): ToeplitzStatus {.raises: [], meter.} =
+  ## MSM per position, then IFFT
```

**Issue:** **Docstring is a single fragment, not a complete description.**

The docstring "MSM per position, then IFFT" is an abbreviated note rather than a proper docstring. It does not explain:

- What "MSM" means (multi-scalar multiplication) for readers unfamiliar with crypto terminology
- That this is the terminal step of the accumulator workflow (called after all `accumulate` steps)
- The `output` parameter description
- That the IFFT is performed in-place on the output buffer

**Suggested Change:**

```nim
proc finish*[EC, ECaff, F](
  ctx: var ToeplitzAccumulator[EC, ECaff, F],
  output: var openArray[EC]
): ToeplitzStatus {.raises: [], meter.} =
  ## Complete the accumulation: perform MSM per output position, then IFFT.
  ##
  ## After `L` calls to `accumulate`, this computes the final result by:
  ## 1. For each output position i, performing multi-scalar multiplication
  ##    (MSM) of the L accumulated scalar-point pairs.
  ## 2. Applying inverse FFT in-place to produce the time-domain output.
  ##
  ## @param output: Result buffer (EC points, length = size). IFFT is in-place.
  ## @return: Toeplitz_Success, or Toeplitz_MismatchedSizes if offset != L
```

## Positive Changes

1. **Excellent `bit_reversal_permutation` documentation** — The refactored `bit_reversal_permutation` function in `fft_common.nim` has clear, well-structured docstrings for both the aliasing-safe and `noalias` variants, explaining the performance trade-off and aliasing detection behavior.

2. **Comprehensive `transpose.nim` documentation** — The new `transpose.nim` module includes benchmark results, element-size-specific block size recommendations, performance figures, and links to the benchmark file. This is a model of practical documentation.

3. **Mathematical invariant comments in polyphase decomposition** — The new comment at line 292-294 of `kzg_multiproofs.nim` provides a rigorous mathematical explanation of why `j < 0` after the loop, with the invariant derivation (`CDS*L == 2*N`). This is the kind of comment that makes cryptographic code auditable.

4. **Consistent error message prefix** — Error messages in `kzg_multiproofs.nim` use the "Internal error:" prefix consistently across all `doAssert` messages, making it easy to distinguish internal invariant violations from expected error conditions.

5. **`ToeplitzAccumulator` field-level documentation** — Each field of the `ToeplitzAccumulator` object has an inline comment explaining its purpose and size, including the important note about user-owned descriptors not being freed by `=destroy`. This is excellent for a complex data structure.

6. **`computePolyphaseDecompositionFourier` doc update** — The docstring was updated to document the new step 3 ("Batch convert Jacobian to affine") and the `@param` was updated to note "affine form", accurately reflecting the API change.

7. **`toeplitzMatVecMul` docstring updated for new implementation** — The function's docstring was updated to describe the new `ToeplitzAccumulator`-based algorithm instead of the removed `toeplitzMatVecMulPreFFT`, keeping documentation in sync with implementation.
