## Findings

### [SECURITY] SEC-A-001: Variable-time batch affine conversion in IPA proving path — secret-dependent z-coordinates exposed to data-dependent branching

**Location:** constantine/commitments/eth_verkle_ipa.nim:225-264,649
**Severity:** Medium
**Confidence:** 0.5

**Diff Under Review:**
```diff
-    lrAff.batchAffine(lr)
+    lrAff.batchAffine_vartime(lr)
```
```diff
-    batchAffine(
+    batchAffine_vartime(
       gL.asUnchecked(),
       gg[].asUnchecked(),
       gL.len
     )
```
```diff
-    tmpAff.batchAffine(tmp, affNeeded) # TODO: introduce batchAffine_vartime
+    tmpAff.batchAffine_vartime(tmp, affNeeded)
```

**Issue: Variable-time batch affine conversion on potentially secret-dependent EC coordinates in IPA proving path.**

The `ipa_prove` function (Inner Product Argument proving) converts EC points from projective/Jacobian to affine form using the newly introduced `batchAffine_vartime` instead of the constant-time `batchAffine`. The intermediate EC points `lr`, `gg`, and the aggregated commitment points in `sumCommitmentsAndEvalsByChallenge` are derived from secret polynomial coefficients (`poly` parameter) through:

1. Pedersen commitments: `pedersen_commit(lr[0], aR)` where `aR` comes from the secret polynomial
2. Scalar multiplications: `aRbL_Q.scalarMul_vartime(aRbL, Q)` where `aRbL = innerProduct(aR, bL)` involves secret coefficients
3. Accumulated sums: `lr[0] ~+= aRbL_Q` — z-coordinates of these sums depend on the secret polynomial values

The `batchAffine_vartime` implementation uses data-dependent branching on z-coordinate values:
```nim
zero(i) = SecretWord projs[i].z.isZero()
if zero(i).bool():
    affs[i].setNeutral()  # skip inversion for this element
else:
    # perform field multiplication and coordinate conversion
```

This means the execution time depends on which z-coordinates are zero, and z-coordinate values are influenced by the secret polynomial coefficients.

**Attack Scenario:** An attacker with side-channel measurement capability (e.g., power analysis, cache timing, or precise cycle counting) on the prover's machine could potentially recover information about the z-coordinates of intermediate EC points, which are derived from the secret polynomial. This requires:
- Physical or very close proximity to the prover (not network-level)
- Repeated measurements with controlled inputs
- Sophisticated statistical side-channel analysis

**Mitigating Factors:**
- The IPA protocol's soundness does NOT depend on constant-time proving — the proof is public and verified separately
- Z-coordinates being exactly zero is rare in normal IPA execution (requires specific cancellation)
- The `scalarMul_vartime` calls earlier in the same function already use variable-time operations
- IPA proving typically runs in trusted environments (not adversarial hosting)

**Suggested Change:** If constant-time proving is a security requirement for this deployment, revert these three calls to use `batchAffine` (constant-time version). Alternatively, document the timing side-channel explicitly if performance is prioritized.

## Positive Changes

- **Error handling hardening in `computeAggRandScaledInterpoly`:** The function now uses `doAssert` with descriptive error messages instead of silently returning `false`. Since all inputs are public verification data (commitments, evaluations, column indices), crashing on invalid input is acceptable and arguably more debuggable than silent failure.

- **Memory safety improvements:** The new `ToeplitzAccumulator` has proper `=destroy` semantics with nil-checks, defensive double-init handling, and explicit `=copy` prohibition. The `check`/`checkReturn` templates provide structured error handling with proper resource cleanup.

- **In-place FFT optimization with aliasing safety:** The `bit_reversal_permutation` function now detects dst/src aliasing and uses a temporary buffer when needed, preventing data corruption during in-place FFT operations.

- **`Alloca` tag removal from FFT functions:** Removing `Alloca` tags from EC FFT functions that no longer use stack-allocated temporaries (due to iterative implementations) is correct — it prevents static analyzers from falsely assuming large stack usage.

## Additional Observations (Non-Findings)

1. **`batchAffine_vartime` in `ec_scalar_mul_vartime.nim` (lines 362, 430):** The `scalarMul_wNAF_vartime` and `scalarMulEndo_wNAF_vartime` functions already have `{.tags:[VarTime]}` annotations — using `batchAffine_vartime` internally is consistent. These process precomputation tables derived from public base points (not secret-dependent), so the timing behavior is expected.

2. **`batchAffine_vartime` in KZG verification (`kzg.nim:363`, `kzg_parallel.nim:170`):** Verification processes public commitments and evaluation points — no secret-dependent data flows through the batch affine conversion.

3. **`batchAffine_vartime` in `kzg_coset_prove` (`kzg_multiproofs.nim:458`):** The polynomial coefficients are typically public blob data in PeerDAS context. The proof generation timing is not security-critical.

4. **`multiScalarMul_vartime` in `ToeplitzAccumulator.finish` (`toeplitz.nim:295`):** The MSM scalars are derived from public polynomial coefficients, and the points come from the public SRS. No secret-dependent timing.

5. **N <= 0 guard in all `batchAffine_vartime` functions:** All four new vartime variants (ShortWeierstrass projective, ShortWeierstrass Jacobian, TwistedEdwards projective) correctly guard against empty batches, preventing potential division-by-zero or out-of-bounds access.
